# Starmap Enterprise Catalog Control Plane

Last updated: 2026-07-09

This is the durable execution control plane for making Starmap the authoritative
LLM catalog for Starport. It converts the architecture review into ordered work
with explicit evidence, regressions, and completion gates.

## Mission

Make Starmap a reliable, auditable, provider-aware catalog engine and
distribution system for Starport, an open-source enterprise OpenRouter-style
product written in Go.

The target provides:

- correct model-definition and provider-offering identity;
- immutable, versioned catalog generations;
- transactional persistence and publication;
- explicit, resilient source observations;
- field authority that prefers valid provider pricing when present;
- durable scheduling above an idempotent sync transaction;
- reproducible artifact and hosted distribution;
- graceful handling of upstream schema drift;
- deterministic and live verification suitable for enterprise trust.

## Baseline

The 2026-07-09 working-tree review found strong foundations: sync orchestration
is concentrated, provider concurrency is bounded, reconciliation rules are
centralized, ownership and query modules have focused tests, and `make verify`
passes.

It also reproduced production-contract failures outside that gate:

- regular CLI update applies before confirmation;
- filesystem save leaves removed model files behind;
- catalog copy shares mutable configuration;
- read-only catalog exposes mutable collections;
- concurrent auto-update lifecycle calls race and panic;
- flattened model identity loses provider-offering identity;
- remote update calls a route the server does not expose;
- full-catalog copy-on-request costs milliseconds and megabytes per request.

The second review found an active source outage: the live models.dev schema no
longer fits Starmap's parser, so the default HTTP enrichment source aborts. It
also established the distribution and scheduling decisions recorded below.

## Status Legend

- `DONE`: success criteria are proven and recorded.
- `IN_PROGRESS`: active review or implementation.
- `PENDING`: not started.
- `BLOCKED`: cannot continue without external state or user authority.
- `DEFERRED`: deliberately postponed with durable rationale and a user-approved
  scope decision; it is not a completion state.
- `REJECTED`: a finding is intentionally not implemented after the user accepts
  the recorded rationale, risk, and alternative or non-action.

## Global Constraints

1. Preserve unrelated changes in the dirty working tree.
2. Work in ledger order unless a discovered dependency is recorded.
3. Add a characterization or regression before each reproduced fix when practical.
4. Library, server, and scheduler paths must never prompt on stdin.
5. Never log or persist credentials or reusable credential fingerprints.
6. Use typed errors; do not panic for source, input, persistence, or serialization failures.
7. Publish a catalog generation only after validation and durable commit.
8. Report deterministic, live-provider, hosted-CI, and distribution gates separately.
9. Compatibility changes require a migration path and old-format fixture.
10. Coverage alone never proves enterprise readiness.
11. Every finding class is in scope: blocker, reliability, cleanup, and
    nice-to-have. Classification affects order and risk, not whether the finding
    must receive a terminal disposition.
12. No finding may disappear through task consolidation, renaming, refactoring,
    or supersession. A superseding task must retain an explicit mapping and prove
    evidence equivalent to or stronger than the superseded success criteria.
13. `DEFERRED` requires durable rationale, explicit residual risk, a re-entry
    condition, and a user-approved scope decision. Agents may not defer work only
    to make the plan appear complete.
14. Control-plane completion requires every finding row to be `DONE` or
    user-accepted `REJECTED`; phase/task status alone is insufficient.
15. Keep deterministic, race, live-provider, migration, hosted, artifact, and CI
    evidence as distinct named results. One evidence class cannot substitute for
    another.

## Second Review Findings and Decisions

### Current data and source sanity

- The embedded tree contains 761 files and is about 3.3 MB. It validates as 11
  providers, 83 authors, and 730 provider model records, but has no generation
  manifest or freshness declaration.
- Embedded provenance timestamps and the latest source-control changes to the
  catalog are from October 2025. The embedded models.dev payload is from
  September 2025.
- On 2026-07-09, the live models.dev response was 3,020,966 bytes with 154
  providers and 5,427 models. The embedded fallback was 268,114 bytes with 44
  providers and 669 models.
- The live endpoint returns an ETag and revalidation cache policy, but the HTTP
  adapter uses file age, does not send `If-None-Match`, and records no revision.
- A live dry-run failed because `experimental.modes.provider.body` now contains
  nested objects while the parser requires `map[string]string`. The syntactic
  precheck promoted the incompatible response before typed parsing, so there was
  no last-known-good parsed fallback.
- Both models.dev source adapters use package-level `sync.Once`. A long-running
  scheduler cannot observe later source changes, and the first configured cache
  directory becomes process-global.
- The default source set runs both HTTP and Git/build models.dev adapters. They
  should be alternatives: HTTP is the production input and a commit-pinned Git
  build is an explicit verification path.
- Provider source output conflates configured bootstrap providers, credential
  skips, successful live observations, and partial provider failures. One
  provider fetch error currently aborts the complete source pipeline.
- Provider source-shape fixtures are from August-October 2025 and cover six
  provider families. They verify known fixtures but do not detect live unknown
  paths or record observation completeness.
- `make embed` downloads directly over the target file without HTTP failure
  checking, checksum, typed parse validation, or atomic promotion. Its later
  `starmap validate` invocation prints help rather than running catalog
  validation, and two update invocations use obsolete CLI flags.

### Authority decision

Provider pricing is authoritative only for the provider offering that emitted
it, and only after unit, currency, non-negativity, effective-time, and structural
validation. models.dev pricing is the fallback for missing or rejected provider
prices. A curated override is allowed only with explicit evidence, rationale,
and an effective interval.

This requires removing two independent models.dev-first mechanisms: the field
authority table and the special complex-pricing merge. Price is not a canonical
model-definition field; it belongs to a provider offering. Zero and missing are
distinct states. The winning observation and every rejection/fallback reason
must remain queryable.

Initial authority outline:

| Attribute | Primary | Fallback | Important rule |
| --- | --- | --- | --- |
| Offering price | Provider observation | models.dev, then curated override | Provider-specific, semantically valid, freshness recorded |
| Offering availability and provider ID | Provider observation | Last-known-good with degraded flag | Absence in a partial response is not deletion |
| Provider limits and regions | Provider observation | models.dev | Merge only absent subfields; retain evidence |
| Explicit provider capabilities | Provider observation | models.dev | Known `false` is not treated as empty |
| Definition lineage and authorship | Curated local definition | models.dev | Provider labels are evidence, not automatic identity |
| Description and presentation metadata | Curated local definition | models.dev, then provider | Content policy is explicit |
| Routing preference | Starport policy | None | Not a catalog ingestion fact |

### Distribution decision

Use a hybrid, generation-based design:

1. Keep one known-good embedded generation for offline and air-gapped startup.
2. Publish each validated generation as an immutable GitHub Release asset with
   checksum and build attestation. Do not use expiring Actions artifacts as the
   runtime distribution channel.
3. Serve `starmap.agentstation.ai/v1/catalogs/latest` as a small
   latest-compatible manifest and immutable payloads at
   `/v1/catalogs/{generation}` from object storage/CDN.
4. Let Starport pin an exact generation or track latest-compatible according to
   deployment policy. Preserve previous generations for rollback.
5. Optionally mirror the same manifest/payload as an OCI artifact for enterprises
   that require registry-native digest policy and internal mirroring.

| Channel | Role | Benefits | Costs and limits | Version binding |
| --- | --- | --- | --- | --- |
| Embedded generation | Offline and air-gapped last-known-good bootstrap | No network dependency; deterministic startup | Binary growth; freshness is limited to build cadence | Records an exact generation ID, not the binary version |
| GitHub Release asset | Immutable public provenance and reproducible download | Checksums, attestations, durable release history | GitHub availability and enterprise egress policy; unsuitable as a mutable latest pointer | Exact generation ID and payload digest; independent of Starmap release tags |
| Hosted manifest and payload | Low-latency latest-compatible discovery and immutable generation delivery | ETag revalidation, CDN reach, staged promotion, rapid rollback | Requires operated availability, telemetry, SLOs, and trust policy | Manifest selects by schema compatibility; payload URL pins generation ID |
| OCI mirror | Registry-native enterprise mirroring and digest policy | Fits existing admission, replication, and air-gap workflows | Additional publication/retention machinery; optional rather than the canonical latest pointer | Same payload digest and generation ID as release/hosted channels |

The artifact does not need to match the Starmap or Starport release version.
Compatibility is determined by catalog `schema_version` and a declared consumer
compatibility range. `generation_id` is an immutable logical identifier; the
payload and distribution artifact are independently content-addressed. The
manifest includes generation ID, schema version, generated time, payload SHA-256
and size, source revisions/ETags and observation times, completeness/degraded
state, validation version/result, and minimum/maximum supported consumer schema.

Immutable generation URLs use long-lived immutable caching. The latest manifest
uses ETag revalidation and a short freshness lifetime. Consumers verify checksum
and compatibility before atomic activation.

### Scheduling decision

Add scheduled synchronization, but do not hide it in `starmap.New` or a package
goroutine. There are two explicit schedules:

- The Starmap project runs a scheduled generation workflow, initially daily and
  manually triggerable, which fetches sources, validates, produces a diff and
  evidence, publishes only on change, and never replaces the last-known-good
  generation on failure.
- Starport/deployment automation polls the latest-compatible manifest or invokes
  idempotent sync with jitter. HA deployments use a lease/single-flight boundary
  and atomically activate one verified generation.

No schedule can be reliable until source calls are reentrant, catalog commit is
transactional, and publication is generation-based. Those dependencies explain
the ledger order.

   closing a superseded row.

    condition, and a user-approved scope decision. It does not satisfy completion.

    accepts its rationale, risk, and disposition.

    results separate in both the Evidence Log and closeout report.
